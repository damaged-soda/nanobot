# 基于 nanobot 的可验证 Agent 架构迭代提纲（讨论总结）

## 一、摘要

本文档总结了今天讨论的核心判断：现有 nanobot 这类轻量 agent 在长期运行、自动化调度、代码修改等场景中，暴露出的主要问题并不是“模型不够聪明”，而是**状态归属不清、未来执行上下文不可重放、缺少可执行验收、以及没有清理过期修改的机制**。

因此，下一轮架构迭代的重点不应是继续强化 memory 或 prompt，而应引入两个外置的保证层能力：

1. **TDAD（Test-Driven Agent Development / 以规格和可执行测试驱动的 agent 变更）**：要求 agent 不只是“说自己会了”，而是必须通过对未来场景的模拟与验收，证明自己的修改真的能实现目标。
2. **Cleanup Verifier**：要求 agent 在“已经把问题修好”之后，继续证明最终保留下来的每一块修改仍然是必要的，自动清理那些已经失效或多余的改动。

这两个能力的关键不是塞进 agent 内部，而是作为一个**外部的 assurance layer（保证层）**，套在 nanobot 现有运行时外面。这样既能保留 nanobot 的简洁、松耦合和易改动特性，又能显著提升自动化任务和代码修改任务的可靠性。

---

## 二、背景：今天识别出的两个核心问题

### 1. source-of-truth 错位

典型例子是：用户说“以后某个 cron 工作要注意 xxx”，agent 把这条信息写进了全局 memory，而不是写进该 cron 自身的配置、说明或可执行工件。

结果是：

- 当前对话里看起来 agent 已经“记住了”；
- 但未来 cron 真正运行时，未必会读取这段 memory；
- 于是这条约束在自动执行时失效。

这类问题的本质不是遗忘，而是**写到了错误的状态面（state surface）**。一条约束被保存了，但没有保存到未来执行真正会读取的那个位置。

### 2. 过期修改残留

另一个高频问题是：agent 修改代码时经常先改 A，后来发现方向不对，又去改 B，最后通过测试的真正原因是 B；但 A 的修改却留在代码里，没有被清掉，成为长期技术债的一部分。

结果是：

- 代码虽然“暂时过了”；
- 但最终 diff 中混入了已经不再必要的修改；
- 长期积累后，系统会越来越脏、越来越难维护。

这类问题的本质不是单次修复失败，而是**agent 不会撤销过期假设**。

---

## 三、根因判断

这两个问题表面不同，但从架构上看，根因是一致的。

### 1. 状态 ownership 不清

系统里往往同时存在多种可写状态：

- 全局 memory
- 用户偏好
- 项目级规则
- job / cron 级别的配置
- 运行历史
- 工作区文件 / 代码

如果这些状态之间没有明确 owner，agent 就会把信息写到“最方便写”的地方，而不是“未来最应该读”的地方。

### 2. 未来执行上下文不可重建

agent 经常是在**当前对话上下文**里判断自己“应该没问题了”。

但很多任务真正重要的是：

- 明天 9 点运行的 cron 看不看得到这条约束？
- 重启后会不会重复发送提醒？
- 修改后的模块在未来独立执行时还能不能成立？

如果系统无法重建“未来实际执行时的上下文”，那 agent 的“我记住了”就没有可验证意义。

### 3. 验收标准不是可执行契约

很多 agent 只是根据自然语言判断“应该做完了”，而不是基于一组可执行、可重放、可断言的成功条件。

例如提醒任务真正的成功条件应当是：

- 08:59 不发送；
- 09:00 正好发送一次；
- 重启后不重复发送；
- 删除任务后不再发送。

这是一个**可执行的时序契约**，而不是一句模糊的自然语言承诺。

### 4. 缺少“最小必要修改”目标函数

多数 agent 的停止条件只有一个：**“问题现在过了吗？”**

但缺少第二个问题：

**“为了让问题通过，最终保留下来的每一块改动都还是必要的吗？”**

没有这一层，agent 就会不断向系统中沉积历史试错痕迹。

---

## 四、设计目标

这次架构迭代的目标，不是推翻 nanobot，而是在保留其优点的前提下，为它补上“可验证”和“可清理”的能力。

核心目标有五个：

1. **保留 nanobot 的简洁、松耦合、易改动特性。**
2. **让未来能否正常工作变成可执行验证，而不是记忆承诺。**
3. **让所有持久化修改先在隔离环境中完成，再通过验证后发布。**
4. **让最终变更集尽量最小，只保留必要修改。**
5. **缩小 memory 的职责，避免它变成配置和控制平面的垃圾桶。**

---

## 五、核心设计原则

### 1. Artifact 优先于 Memory

凡是会影响未来自动执行的约束，都应该落到真正的 source of truth artifact 上，而不是仅存在于全局 memory 中。

换句话说：

- memory 适合放检索辅助信息、非关键事实、总结性知识；
- project rule 应该放在 project rule 的载体中；
- job-specific constraint 应该放在 job spec / job instruction 中；
- 代码行为应该由代码与测试共同定义。

### 2. 一条约束必须有唯一 owner

系统里每类约束都应能回答一个问题：

**“到底哪个工件对这条行为负责？”**

如果回答不出来，就说明这条约束还没有被正确建模。

### 3. 先模拟未来，再发布现在

agent 的修改不能只在当前对话里“看起来对”，而应先在一个能够模拟未来调用场景的环境中运行，通过验收后才能发布。

### 4. 优先使用外部 verifier，而不是自我反思

真正可靠的不是“让同一个模型再想一遍自己对不对”，而是：

- 程序化检查
- 可执行测试
- 环境模拟
- 事件断言
- 回归套件
- 差分验证

模型可以参与解释和规划，但最终应由外部 verifier 决定是否可以发布。

### 5. 通过验收不够，还必须清理多余改动

系统的成功标准应由两个部分共同组成：

- **Sufficiency**：目标确实达成；
- **Necessity**：最终保留的修改仍然都是必要的。

只有同时满足，才允许发布。

### 6. Scratch first, publish later

所有具有持久副作用的变更，应先发生在一个隔离的 scratch 环境中。只有当验证通过后，变更才被提升到真实 workspace 或正式状态中。

---

## 六、建议中的最小改动架构

### 1. 总体思路：在 nanobot 外面加一圈 assurance layer

不重写 nanobot 的核心运行时，而是在其外部增加一个很薄的“保证层”。

其职责不是替代 nanobot 思考，而是负责：

- 承接 agent 产生的候选变更；
- 在隔离环境中重放未来场景；
- 验证变更是否满足目标；
- 清理不再必要的修改；
- 只有全部通过时才允许发布。

架构关系可以概括为：

```text
请求 / 用户意图 / 自动任务
        ↓
   nanobot runtime
（负责规划、编辑、执行）
        ↓
   scratch environment
（隔离工作区 / 隔离状态）
        ↓
   verification harness
   ├─ TDAD verifier
   └─ cleanup verifier
        ↓
     publish gate
        ↓
    real workspace / real state
```

### 2. 为什么这是“最小改动”

因为它不要求把 verifier 塞进 nanobot 的脑子里，也不要求把整个 runtime 推倒重写。

它只要求建立一个明确边界：

- nanobot 继续负责“生成候选方案”；
- assurance layer 负责“证明这个方案可以被发布”。

这样一来：

- nanobot 本身依然可以保持轻量；
- 验证逻辑不会污染主循环；
- 新能力可以逐步扩展，而不是一次性重构全系统。

---

## 七、TDAD 在这个架构中的位置

### 1. TDAD 的实用化理解

在这里，TDAD 不必一开始就追求论文版的完整形态。它的最小可用意义是：

**把任务目标编译成一组可执行的验收条件，然后让 agent 的修改必须通过这些验收。**

也就是说，系统不再接受这种回答：

- “我已经记住了。”
- “我已经设置好了。”
- “理论上应该会工作。”

而必须接受这种回答：

- “我在模拟的未来场景中验证过，满足以下条件，因此允许发布。”

### 2. TDAD 的核心作用

TDAD 主要解决的是 **“修改是否真的支持未来目标”**。

它适合最先应用在以下任务上：

- cron / reminder / scheduling
- 具备明确输入输出的 automation
- 具备确定性测试条件的代码修改
- 可以通过 mock / replay / sandbox 验证的任务

### 3. 对 cron / reminder 场景的意义

比如“每天 9 点发送提醒”这类任务，真正应该被验证的是：

- 在 08:59 是否不发送；
- 在 09:00 是否发送一次；
- 重启后是否不会重复发送；
- 停用后是否不再发送。

一旦这些都能被模拟执行并断言，agent 就不再依赖“记忆自己应该怎么做”，而是把行为交给了一个明确、可验证的执行契约。

### 4. 关键限制

TDAD 要求一个非常重要的前提：

**被修改的必须是未来运行时真正会读取的工件。**

如果 agent 把 job 级约束写进 global memory，而 future execution 根本不看这块 memory，那么即使当前对话里测试看起来通过，这也是一个假的成功。

所以 TDAD 必须和 source-of-truth discipline 一起落地。

---

## 八、Cleanup Verifier 在这个架构中的位置

### 1. Cleanup Verifier 解决什么问题

Cleanup Verifier 解决的是另一个常见但常被忽视的问题：

**agent 修复成功后，如何清理掉那些已经不再必要的历史修改。**

它不是为了“找错误”，而是为了确保：

- agent 的最终 diff 尽可能小；
- 已经过期的试错改动不会被永久保留；
- 系统不会因为 agent 的多轮尝试而越来越脏。

### 2. 它的工作逻辑

一个简化后的流程是：

1. 先让 agent 在 scratch 环境里把目标做成；
2. 确认验收条件已经通过；
3. 对最终 diff 做逐块回退实验；
4. 若撤销某块修改后系统仍然通过验收，则说明这块修改已无必要；
5. 删除这块修改，继续迭代；
6. 直到最终 diff 中每一块改动都“删不得”为止。

### 3. 它的架构意义

这一步实际上给 agent 增加了一个新的发布标准：

不只是“做成了”，还要“做成得干净”。

这会显著改变系统长期演化的方向：

- 没有 cleanup verifier 时，agent 趋向于堆积技术债；
- 有 cleanup verifier 时，agent 会在发布前自动回收过期试错痕迹。

---

## 九、建议中的状态模型

如果要让上述机制长期成立，需要对状态做一个简化但明确的分层。

一个足够实用的划分可以是：

### 1. 用户长期偏好

负责持久化与“这个用户是谁、长期偏好是什么”有关的信息。

### 2. 项目级规则

负责定义当前 workspace / 项目 / repo 范围内的长期约束。

### 3. Job / Task 级规格

负责定义某个自动化任务、某个 cron、某个具体作业的行为要求。

### 4. 运行时临时状态与日志

负责记录某次执行发生了什么，但不承担规范未来行为的职责。

### 5. 工作区工件 / 代码 / 配置

负责承载那些最终真正决定系统行为的内容。

关键纪律是：

- job-specific constraint 不能只存在于全局 memory；
- project-level rule 不能漂在某次聊天中；
- run history 不能冒充配置；
- memory 不能作为唯一的控制平面。

---

## 十、为什么这条路适合 nanobot

nanobot 的价值在于：

- 结构相对简洁；
- 模块边界还算清晰；
- 不像一些“大一统” agent 平台那样沉重；
- 更容易在外围套入验证和发布机制。

这意味着它特别适合走“外置 assurance layer”的路线，而不是朝“把所有逻辑都塞进主 loop”继续膨胀。

从架构风格上看，最值得保留的是：

- runtime 继续轻量；
- 能力通过边界清楚的模块扩展；
- 验证与执行解耦；
- 变更通过 publish gate 统一管理。

如果未来某些内部点位需要轻微调整，也应遵循这个原则：

**尽量增加少量、稳定、通用的扩展点，而不是把一整套复杂验证逻辑侵入主循环。**

---

## 十一、建议中的迭代方向

本文档不细化到文件和实现层，只给出适合作为后续迭代计划基础的方向。

### 第一阶段：先做“可验证的自动化任务”

优先选择最适合 TDAD 的场景，例如：

- cron
- reminder
- schedule-based automation

目标是先打通：

- source-of-truth 的归属纪律；
- scratch 执行；
- mock 时间 / 模拟未来执行；
- 发布前验收。

### 第二阶段：引入 cleanup verifier，覆盖代码修改任务

当基础验证链路稳定后，再将其推广到代码修改任务中：

- 所有代码修改先在隔离环境完成；
- 验收通过后执行 diff 最小化；
- 只有必要改动才能进入正式 workspace。

### 第三阶段：统一 assurance framework

在前两个阶段基础上，将以下内容逐步统一：

- 规格表示
- 验收执行方式
- 观察与证据收集
- 发布门禁
- 回归测试
- 隐藏测试 / mutation testing / spec evolution 等增强能力

这样 TDAD 和 cleanup verifier 就不再是两个孤立功能，而会成为同一个保证层中的两个标准能力。

---

## 十二、成功标准

这轮架构迭代是否成功，可以用几个很直接的问题来判断：

### 1. 对任何一条影响未来行为的约束，系统都能回答：

**“它的 owner artifact 是什么？”**

如果回答不出来，说明建模仍然有问题。

### 2. 对任何一个自动化任务，系统都能回答：

**“我如何模拟未来执行，并验证它真的会成功？”**

如果只能回答“agent 说它会记住”，那说明仍然不合格。

### 3. 对任何一次代码修改，系统都能回答：

**“最终保留下来的每一块 diff 是否都仍然必要？”**

如果没有经过 cleanup verifier，这个答案通常是不可信的。

### 4. 系统的长期演化方向应当发生改变：

- 约束不再漂在 memory 中；
- 自动任务不再依赖“下次 agent 会不会想起来”；
- 代码修改不会持续积累历史试错残留；
- 整体系统会越来越可审计、可回放、可维护。

---

## 十三、非目标与边界

为了避免期望失控，需要明确这轮架构迭代的非目标：

### 1. 不是形式化证明一切任务

很多开放式任务无法在短期内被完整形式化，也不适合一开始就追求“完全证明”。

### 2. 不是让 agent 从此不犯错

目标不是消灭模型错误，而是让错误变得：

- 更容易暴露；
- 更容易被阻断发布；
- 更不容易沉积成长期债务。

### 3. 不是要求所有场景都立刻 TDAD 化

应先从最可验证、最有收益的任务类型开始，再逐步扩展。

### 4. 不是用 verifier 取代 runtime

nanobot 仍应是执行与规划的主体；verifier 是发布门禁与保证层，不应反客为主变成一个新的“大而全”内核。

---

## 十四、结论

今天讨论形成的核心共识可以归纳为一句话：

**问题不在于 agent 是否“记得住”，而在于系统是否要求它把约束写到正确的工件上，并在发布前证明这些工件真的支持未来目标。**

因此，这轮迭代的重点不应是更强的 memory，而应是：

- 更清楚的状态归属；
- 更可执行的目标规格；
- 更可重放的未来场景模拟；
- 更严格的发布前验证；
- 更自动化的多余改动清理。

对 nanobot 而言，最合适的路线不是大规模重写，而是在保留其轻量 runtime 的前提下，加上一层外置的 assurance layer，让系统逐步从“agent 口头承诺”过渡到“agent 只有在通过可执行验证后才能发布”。

这将为后续的详细迭代计划提供一个清晰的架构方向。
